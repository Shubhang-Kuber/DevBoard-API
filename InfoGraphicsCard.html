ANATOMY OF A REST API: A SCIENTIFIC CUTAWAY
How an HTTP request travels through DevBoard API — from client to database and back, layer by layer, with every mechanism exposed.
A
🔐
Authentication Layer

Two-step identity system separating how passwords are stored from how sessions are proven.

bcrypt (registration): Applies a one-way hash with a cost factor, so even if the database leaks, passwords cannot be reversed. The salt is embedded in the hash string.

JWT (every request): The server signs a JSON payload with a secret key. The client stores this token and sends it in Authorization: Bearer <token>. The server re-verifies the signature — no session table needed.

  • Stateless — no server-side session store
  • Token contains userId as payload
  • Expiry enforced on every decode

Middleware Chain

Express processes each request through a pipeline of functions. Each calls next() to pass control forward, or throws to the error handler.

json() → cors() → auth.js
→ router → controller
→ errorHandler()

  • auth.js — verifies JWT, attaches req.user
  • Global error handler catches anything thrown in the chain

C
🛡️
Ownership Authorization

Authorization ≠ Authentication. Auth confirms who you are; ownership checks confirm what you're allowed to touch.

Every query appends AND user_id = ? bound to req.user.id. Even if a user guesses another user's task ID, the SQL returns no rows — the API responds 404, not a data leak.

  • No client-supplied userId is ever trusted
  • Token payload is the only source of identity
  • Prevents IDOR (Insecure Direct Object Reference)
D
⚠️
Centralized Error Handling

Express's 4-argument error handler (err, req, res, next) is registered last. Any throw anywhere in the chain lands here.

Produces a consistent {"error":"…"} envelope instead of stack traces leaking to clients. Status code is carried on the thrown Error object.

  • 400 — validation failures
  • 401 — missing / invalid token
  • 403 — wrong owner
  • 404 — not found
  • 500 — unexpected server error

Why junction tables? A task can have many tags; a tag can belong to many tasks. A single FK on either table would enforce a one-to-many constraint. Junction tables (task_tags, bookmark_tags) hold only two FK columns — one pointing to the resource, one to the tag — making the many-to-many relationship explicit and queryable.

Shared tags: The same tags table serves both tasks and bookmarks. A tag named "typescript" is one row — both resource types reference it via their own junction table.

Design Trade-offs & Decisions

🔀 PATCH vs PUT

PATCH (used here) sends only the changed fields. PUT sends the full replacement object — omitting a field deletes it. PATCH is safer for partial updates.

🌐 REST vs GraphQL

REST maps resources to URLs; each endpoint has a fixed shape. GraphQL lets clients request exactly the fields they need — more flexible, more complex to implement.

🗄️ SQLite vs Postgres

SQLite is file-based, zero-config, single-writer. Postgres handles concurrent writes and scales horizontally. SQLite is ideal for learning and single-server apps.

🔑 JWT vs Sessions

JWT is stateless — no DB lookup per request. Sessions are revocable but require server-side storage. JWT trades revocability for simplicity and scalability.
